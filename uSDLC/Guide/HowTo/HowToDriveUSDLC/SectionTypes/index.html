Section Types

Section types provide clues to readers and writers for what the section and often the child pages are to provide. The uSDLC tasking system also refers to section types for some actions. Guide sections that aren't referenced require attention. A feature section without scenarios is incomplete. For the rest, use them (or not) as it suites your project.
  1. Project: root page for a project or sub-project
  2. Vision: an expansion of a vision point
  3. Goal: goals are specific requirements to meet a vision point
  4. Capability: objectives required to meet a goal
  5. Feature: product functionality to implement a capability
  6. Scenario: examples of how a feature will be used
  7. Script: computer programs
  8. Data: required domain information for scripts and scenarios
  9. Guide: documentation fragment
  10. Page: untyped child page
  11. General: untyped section
  12. Issue: Item of unknown result to add to a register
Guide

Project

In design by decomposition, a project (or sub-project if you like) can start anywhere in the parent project. It is autonomous - and most commonly new functionality being added to an existing project.

Guide

Vision

Every new project starts with a vision. It does not need to be detailed. A set of bullet points is often a good starting place. Depending on the detail of the vision, these points can be further vision pages or high level goals.

In design by decomposition, visions can occur at any level of the project. When adding new functionality to a previously completed project, the list of benefits constitutes a vision.

Guide

Goal

Goals stem from the vision, but may extend past the vision to other stake-holders. Goals should:

  • Cover long time spans relative to capabilities and objectives
  • Can be intangible or tangible
  • should be people oriented
  • should not predetermine the details of program activities
  • should be expressed as a desired outcome or condition to be achieved

Examples could be:

  • To make required data entry easy.
  • To reduce customers who abandon the ordering process.
Guide

Capability

Goals require capabilities or objectives to be met. For the goal 'To make required data entry easy', one capability may be 'Users might need the capability to copy existing records'.

Guide

Feature

To deliver capabilities, we design features.

As a customer, I want to be able to copy an old order so that I can enter a new order more easily.

Guide

Scenario

To explore and flesh out a feature we provide as many examples as we can in the form of scenarios. These scenarios can be phrased in natural language, but are at the same time computer readable. Scenarios provide the feedback loop across the SDLC. A scenario exercises the production code as it becomes available and will tease out disconnects between vision and produciton. In traditional parlance a scenario can be an acceptance test, integration test or unit test.

Script

Scenarios will need backing actions.

Guide

Data

Display reference files. These are often data files used by instrumentation or example code for the application. They are not executed during instrumentation.
