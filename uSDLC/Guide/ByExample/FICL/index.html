FICL

Fast Interactive Command Language

FICL is a general library for use within other applications. The acronym stands for Fast Interactive Configuration Language. Most current applications using compiled code use static configuration files as name-value pairs or XML. At the high end, larger application use interpreted languages such as BASIC or Groovy. FICL is designed with a small footprint to provide similar facilities for less endowed environments - such as application servers and mobile devices.

For this example we will follow Paul - a contract developer asked to implement an Android application that requires some relative complicated workflow, combinations of which will not be known until runtime. The traditional solution is to provide the user with a fixed set of options. With a new application it is not always possible to know which options are needed. The options are to provide a fixed configuration that covers possible options or the ability to add new workflow combinations to the mobile device as their need is discovered. Paul chose to implement FICL so that new workflows can be sent to the mobile device as needed.

Aside: FICL was developed to provide an optional uSDLC configuration and command structure for uSDLC instrumentation and testing.

Guide

What is a contract developer?

For the purposes of this exercise, a contract developer is one or two developers paid to provide a solution. If they are conracted to join or create a team, they fit into another category.

Guide

Contract Developer Strengths

Contract developers are engaged to bring a project to fruition that the client does not otherwise have the resources to complete. In this role a contract developer provides some compelling advantages:

  1. The budget is for a specific project so that the contract developer is less likely to be purlioned or diverted to other work.
  2. It is possible to bring in skill-sets that are not available in-house - iOS is a Java shop, for example.
  3. New perspectives - Contract developers will bring in knowledge and ideas that can foster change.
  4. Experience - you can contract experienced contractors without the overhead of keeping them on staff once the need has been met.
Guide

Contract Developer Weaknesses

  1. They can leave unmaintainable or indecipherable code. Even good code can be difficult to maintain without an understanding of the framework and tools used during development.
  2. They do not have as much domain knowledge as staff. Without on-going access to domain experts, contract developers can lose direction and not produce a viable product.
  3. Clients expect the contract develop to design as well as develop. This can lead to results may not meet expectiation.
  4. Developers in general are notoriously bad at communication and documentation.
Guide

Client responsibilities

  1. Provide the contract developer with vision and goals for the project. Verbal and incomplete has to be OK as the client has contracted a developer because they do not have the resources for vision and design documents.
  2. Provide access to domain experts. Domain experts have their own job to do. Adding a project to this work-load is not popular. It is best if it is possible to have domain experts involved in the vision. This only happens if they can see the immediate benefits for their day-to-day work.
  3. Provide environmental requirements - space, hardware, etc.
  4. Create a feed-back structure that while not onerous, does provide the overview needed to keep the project on track. This can be a fortnightly update meeting, but it needs to be interactive so that client and developer can be reasonably confident they are seeing the same outcomes.
Guide

Contract Developer Responsibilities

  1. Keep the client informed.
  2. Keep the domain experts involved.
  3. Provide timely results. Use iterative developent and agile practices as a strength, not an excuse.
  4. Provide a product to customer expections with quality and reliability.
  5. Provide sufficient documentation and structure for support and further development at a later time. Documentation needs to include how to set up development and test environments as well as all the other details required to support the project.
Guide

Contract Developers and uSDLC

Owner developers can treat uSDLC as a TDD (Test Driven Development) platform. Contract developers do not have that luxury. They have clients and other stake-holders who must be kept in the loop if the project is to be successful. uSDLC provides

  1. A browser based client so that stake-holders can review and provide feedback without interrupting the project flow.
  2. A framework for "Design by Example" or BDD (Behaviour Driven Development) design principles so that design and design decisions are transparent - both for stake-holders and later for support and additional development.
  3. The simplest way to make the smallest possible amount of documentation clear for stake-holders and project members.
  4. A feedback machanism that closes the loop from vision through design to implementation. Stake-holders can see and even create examples that exercises production code and shows that capabilities and features are implemented as requested.
